# 如何进行分层上下文工程 (Context Engineering)

AI 生成幻觉和项目级翻车的核心原因多为“错误塞入了垃圾或过期的文件上下文”，因此通过精细的文件依赖投喂与回收机制控制 Token 质量至关重要。

### 1. 强力推行三层上下文分级 (Hierarchical Context)
彻底摒弃向 AI 盲目投喂巨型目录的散漫操作。严格在物理或逻辑层执行分层：
*   **Tier 1 全局宪法层 (常驻)**：包含核心架构约定与极严苛底线（如 `.cursorrules`）。全局免死金牌，无论何时必须生效。
*   **Tier 2 瞬态手术台 (高频换出)**：仅包含当前变更的 2-3 个核心强关联文件。利用 `@` 符号精准提取，严禁一键全部读取（@Codebase）。
*   **Tier 3 备用字典层 (按需挂载)**：各类第三方库的 API 文档、旧项目源码片段。除非进行底层基建级大修，平常严格隔离，避免其产生记忆稀释。

### 2. 切断低质量上下文的长尾残留 (Context GC 与新开对话)
上下文内容堆积过长，会导致后续修复严重滞留在初期的错误思考路径之中或产生局部遗忘。建立无情的清场与回收标准：
*   **实施快速弃卷机制 (Fail-fast Session)**：当该条开发链已经过 4 到 5 轮极度复杂的拉锯、打回及重度修改仍未突破防线，其积累的错误试探噪音会彻底摧毁后续质量。立即放弃当前长对话框（相当于执行垃圾回收 Context GC）。
*   **抽取结论提纯换场**：在新对话开展前，命其利用剩余逻辑提纯当下的卡点进度和确切核心根报错信息，拿着这份最高密度的纯净单点结晶重启一个全新的原生空对话环境以全速攻坚。

### 3. 实施主被动防范：构建项目级底层知识树锁 (Knowledge Graph)
越庞大的工程，越需要将项目约定进行解耦切片，使其能够在系统调阅时只获取精准领域内信息。
*   以 `/docs/rules` 结构对细分逻辑进行隔离管控（如区分 `数据库防腐约定.md` 与 `单页前端组件渲染层规划.md`）。指导并强制配置核心规则，要求其涉及修改某区块时只精准引入对应的分片架构约法，用以彻底避开巨文件知识带来的干扰和污染。
