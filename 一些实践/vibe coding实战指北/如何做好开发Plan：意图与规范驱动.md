# 如何做好开发 Plan：意图与规范驱动

Vibe Coding 必须遵循“控制驱动流”：口述核心业务意图 -> AI 生成结构化 plan.md 大纲 -> 人工审定防线红线 -> 审核获准后下放机器运行执行写码。拦截越权狂飙行为。

### 1. 采用意图驱动式下行 (Intent-Driven)
摒弃以具体的代码切入点去指引 AI 拼接语法的途径。使用上界面的组件“状态扭转”或是底层后端高并发的“防重幂等”工程学意图去驱动其产出：
*   **错误指令**：不要具体教其使用特定的某个原生方法处理字符串过滤。
*   **合理指令**：“基于前置的订单流转规则状态机，构建一个具备数据推挽限流并能自我回滚容错的订单队列”。

### 2. 强锁定模块边界契约 (Contracts)
把技术控盘集中在拦截“入参与出参”流向：
*   **框定数据结构**：提前约束和定义完成本次模块接纳的具体对象模型格式，以及其抛出的响应闭环结构。明文警告系统底层禁止模型在演化中脱离原结构，去擅自增加不明逻辑结构字段或是重命名入参流向。

### 3. 在开发前置推入系统验收标准卡点 (抓一头一尾)
在这套极速产出体系中，人工需退守设定系统级卡扣边界作为开发的第一标准。在提出任何一个 Todo 任务的瞬间，必须同步圈定该任务的评价体系，死守“起点意图”与“终点验收”：
*   **抓一头 (业务意图起点)**：明确该 Todo 要解决的核心诉求与数据输入契约，严禁发布无业务锚点的孤立任务。
*   **抓一尾 (量化验收终点)**：强制搭配“最极端的评测限定”（如：“长列表单次滚动耗时>50ms即全案抛弃”），使其成为可执行的测试断言。指令 AI 以该终点准线约束，自主倒逼进行压测与设计重构。
