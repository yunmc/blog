# 如何自动化生成架构决策记录 (ADR)

手动维护 ADR (Architecture Decision Record) 通常被视为繁重任务。而在 Vibe Coding 流水线中，依赖与 AI 推演博弈中留下的大量问答痕迹，这一资产能够转换为零成本自动总结的经验库。

### 1. 利用自然会话提取与收割法 (Session Harvesting)
在与 AI 的高强度争辩纠错（如从状态机瘫痪到最后调通）环节后，保存此高保真过程的对话，拒绝将其随用随弃关闭。
*   **使用提纯密令提取结晶 (Prompt Collector)**：待长代码正式合并过检，即时下发指令收割：“刚刚对于复杂高危支付流转状态的架构攻坚争涉达成防线共识，请基于我两的讨论逻辑流剥离水分，按通用的工程级 ADR 架构，浓缩提纯出一份标准格式记录输出作此后开发判定判例锚点”。

### 2. 规范 ADR 的标准骨架五要素进行约束
必须严防总结日志中掺入杂乱游记等无效成分。强制下发约束骨架填入核心因子论证：
1.  **判定标题核心 (Title)**：点明技术路线核心定论（如“将表单全局管理由 Redux 转为 Zustand”）。
2.  **背景恶劣前提 (Context)**：提取促使其产生技术倒逼变革及迫切重构的痛点因素是什么？
3.  **最终落地形态 (Decision)**：陈述系统最终裁定的技术接盘框架选取及模式定夺。
4.  **排弃参照防线 (Options Considered)**：着重说明在争论中被排斥丢弃的其他竞品方案到底带有怎样不足以支撑该主系统的深层结构性死穴劣势（这是防护其他后来者重复踩坑的最高价值资产）。 
5.  **妥协深坑权衡 (Consequences)**：陈述推行此模式将会落地的积极业务增益，以及附带未能解决、需留存承担何等未来级别的技术欠债。

### 3. 将 ADR 编织入后续 Agent 环境守则系统闭环
记录需要反向服务和约束未来模型的重写过程。
*   **固定归档隔离**：确立核心物理储存边界 `/docs/adr/`，每份决断依照强制时间流约束并加上命名格式编纂 `0001-xxx.md`。 
*   **植入环境锁防线 (System Binding)**：在项目总设根环境配置卡点系统规则，声明并要挟高权限指令：“在发起任何有关订单计算中台组件重构操作起手之前。未经静默全盘阅读调取并匹配核对 `/docs/adr/0002-order-status-lock.md` 深水记录者，一概禁止进行后续任何一行结构改动！”。以此确立防线，避免系统随人员流动而遭到知识降级坍塌。
