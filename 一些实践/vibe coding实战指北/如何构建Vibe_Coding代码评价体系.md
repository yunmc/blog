# 如何构建 Vibe Coding 代码评价体系

AI 具有高算力输出特性，但容易产生幻觉和不合理的逻辑耦合。在 Vibe Coding 模式下，代码审查（Code Review）的重心应转向自动化和防作弊机制。

### 1. 防 AI 作弊的测试驱动开发 (Anti-Cheat TDD)
*   **边界断言前置**：在 AI 编写逻辑前，必须提前提供涵盖异常类型、极限值（如数组溢出、非法格式）的测试用例。要求单测跑通方可进行合并。
*   **高压数据驱动测试 (Data-Driven Testing)**：强制注入海量黑盒数据集，包含极限并发伪造与畸形长字符串。防范其为应对少数正常用例而偷懒硬编码。
*   **严控 E2E 测试越权 (Superedit)**：明确约定 Playwright 等端到端回归测试只允许通过 UI 层操作推进流程，严禁 AI 在测试脚本中直接调用后端 API 修改数据库状态来强行通过测试。

### 2. 禁绝“创可贴式”补丁修复
*   **强制探究 Root Cause**：遇到空指针或类型错误时，要求 AI 先输出推理得到的根本原因，再进行针对性修复。
*   **禁用掩盖型语法**：禁止未经底层梳理直接盲目增加 `as any` 或大范围使用 `try-catch` 包裹报错强行推进任务。

### 3. 多智能体交叉审查流 (Agentic Pull Request Review)
不可由负责开发的同一个 Agent 进行代码检验，需引入多 Agent 交叉审核机制：
1.  **开发提交**：使用负责主攻的 Coder Agent 完成业务模块开发，在本地运行验证后推送 PR 草稿。
2.  **触发安全审查**：配置专精于静态安全、内存泄漏及并发死锁检查的 Review Agent 对更新 Diff 进行强拦截审核。
3.  **多智能体对抗 (Debate Cycle)**：Review Agent 驳回缺陷；Coder Agent 自行修正并重新提交，或在认定驳回不合理时调用指令反驳。
4.  **人工仲裁**：在多方自动妥协且流水线变绿后，最终由人类负责阅读评估并确认 Merge 合并。

### 4. 强制机器主动回测 (Active Grounding)
拒绝单纯的代码文字生成，倒逼 AI 执行真实物理域闭环跑测：
*   **阻断式验证**：禁止 AI 产出完代码后随意自信结束，必须配发强制指令（如“自行终端执行 npm run dev，按次序列出所有你成功渲染的路由”或“去通过 cURL 发起调用证明不报错再交给我”）。 

### 5. 沙盘打样物理隔离验证 (Sandbox PoC)
引入复杂的第三方依赖前，必须在隔离环境中完成试运行：
*   设立独立无干扰的纯净 PoC (Proof of Concept) 项目。
*   由机器在隔离沙盒内部完成新框架与组件的集成操作。无报错后，再作为参照向主业务大盘进行对齐移植。
