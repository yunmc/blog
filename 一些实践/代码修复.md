# 修复策略
## 1. 以 review 为入口，以代码事实为准
review 是线索，不是绝对真理。  
你需要结合当前代码、需求、架构和 diff 判断最合理的修复方式。

## 2. 最小正确修复
优先做：
- 改正确逻辑
- 补齐缺失分支
- 修正状态处理
- 补测试
- 修复接口契约

避免做：
- 无关重构
- 风格清洗
- 大面积命名改造
- 与本次问题无关的结构调整

## 3. 优先修根因
不要只压错误表象。  
例如：
- 空指针问题，优先修输入校验或状态流，而不是到处加判空
- 并发问题，优先修状态一致性或锁粒度，而不是表面补丁
- 测试缺失，优先补能防回归的测试，而不是只补 happy path

## 4. 保持与需求一致
如果 review 建议与 requirements.md 冲突，以需求为更高优先级，但必须在修复说明中明确写出原因。

## 5. 保持与架构一致
如果某种修复会破坏 architecture.md 描述的模块边界、职责或依赖方向，优先选择不破坏架构的修法。

# 遇到 review 意见时的处理规则
对于每一条 review finding，你必须做出明确处理状态：

- fixed：已修复
- partially_fixed：部分修复
- not_fixed：未修复
- rejected：不采纳
- cannot_verify：无法确认

每条 finding 都必须有处理结论，不能静默跳过。

如果某条 finding 不采纳，必须说明：
- 为什么不采纳
- 依据是什么
- 有无替代处理

# 自动化测试门禁要求
每完成一轮代码修复后，必须先执行自动化测试验证修复结果，测试通过后，才允许将本轮修复标记为完成，并写入 `tmp/fix report.md` 或回写到 PR。

## 测试执行原则
1. 必须优先识别项目中已有的自动化测试入口，而不是跳过测试
2. 优先执行与当前改动直接相关的测试
3. 如果能够定位到更小范围的测试（模块测试、单测、集成测试），优先先跑小范围相关测试
4. 在条件允许时，再补充执行更高层级的测试或项目约定的关键测试集
5. 如果测试失败，不允许把本轮修复标记为“完成”
6. 如果测试未执行，不允许把本轮修复标记为“完成”
7. 如果环境限制导致无法运行测试，必须明确说明原因、影响范围和剩余风险

## 必须主动识别的测试信息
修复完成后，你必须主动检查项目中的测试与验证方式，例如：
- 单元测试命令
- 集成测试命令
- e2e 测试命令
- lint / typecheck / build / compile / 静态检查
- 项目文档中约定的验证命令
- CI 配置中的关键校验步骤

你可以通过以下信息源识别测试方式：
- package.json
- pyproject.toml
- Makefile
- justfile
- tox.ini
- pytest.ini
- pom.xml
- build.gradle
- Cargo.toml
- go.mod
- CI 配置文件
- README / 开发文档
- 现有测试目录和脚本

## 测试执行顺序
每轮修复后，按以下顺序执行测试：

1. 先执行与当前修复直接相关的最小测试集
2. 如果最小测试集通过，再执行必要的补充校验，例如：
   - lint
   - typecheck
   - build
   - compile
   - 与改动相关的集成测试
3. 如果项目已有明确规定的必跑测试门禁，则必须满足这些门禁

## 测试结果对修复状态的影响
你必须根据测试结果给出明确状态：

- passed：相关测试已通过，本轮修复可视为完成
- failed：测试失败，本轮修复不得视为完成
- not_run：未执行测试，本轮修复不得视为完成
- blocked：因环境、依赖、权限等原因无法执行测试，本轮修复不得视为完全完成

只有在 `passed` 的情况下，才可以：
- 在修复说明中将对应问题标记为已完成修复
- 回写 PR 表示本轮修复完成

如果状态是 `failed`、`not_run` 或 `blocked`，则必须在修复说明中明确写出：
- 哪些测试执行了
- 哪些测试失败了或未执行
- 失败原因或阻塞原因
- 当前剩余风险

## 测试失败时的处理
如果测试失败：
1. 先分析失败是否由本轮修改引起
2. 如果是本轮修改引起，优先继续修复直到相关测试通过
3. 如果不是本轮修改引起，也必须在修复说明中明确区分：
   - 预存失败
   - 本轮引入失败
4. 不要在测试失败时假装本轮修复已经完成

## 无法执行测试时的处理
如果无法执行测试，必须明确说明：
- 缺少什么环境
- 缺少什么依赖
- 哪些命令无法运行
- 哪些风险因此无法验证
- 哪些问题只能留待下一轮人工或 CI 验证
