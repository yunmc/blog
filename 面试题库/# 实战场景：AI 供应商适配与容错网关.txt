# 实战大作业：AIGC 多厂商适配与弹性调度网关

## 1. 任务背景

在生产环境中，依赖单一 AI 供应商（如 OpenAI, Claude）存在极大的单点风险。由于 API 经常出现限流（Rate Limit）、响应超时或服务宕机，我们需要一个**高可用的网关服务**来屏蔽底层差异，并在极端情况下通过自动重试与故障切换确保业务连续性。

## 2. 核心要求

### 第一阶段：方案调研与架构设计 (Research & Design)

在动手编码前，请在交付文档中包含以下调研内容：

1. **现有方案调研：** 简要调研市面上成熟的 AI Gateway/Proxy 方案（如 LiteLLM, One API, 或 Kong AI Plugin 等）。
2. **优缺点分析：** 对比上述方案在**协议转换、性能损耗、扩展灵活性**方面的表现。
3. **自研方案定义：** 结合本次作业需求，说明你的实现思路及技术栈选型理由。

### 第二阶段：核心功能开发 (Implementation)

你需要交付一个完整的 Package 或服务，实现以下逻辑：

1. **适配层 (Adapter)：** 统一对外接口，屏蔽不同供应商（模拟 A 厂、B 厂）的参数名、鉴权方式及响应结构的差异。
2. **弹性调度 (Resilience)：**
* **指数退避重试：** 遇到 `5xx` 错误或网络抖动时自动重试。
* **自动降级 (Fallback)：** 遇到 `429` (限流) 或重试耗尽时，秒级无感切换至备用厂商。


3. **多维统计 (Observability)：** 实现指标监控，并提供接口或日志输出以下数据：
* **各厂商请求成功率**（Total / Success / Failed）。
* **响应耗时 (Latency)**：统计平均响应时间或 P95 耗时。
* **虚拟成本核算**：基于 Token 消耗量模拟计算（如 A 厂 $0.01/1k, B 厂 $0.02/1k）。



### 第三阶段：自动化测试 (Automated Testing)

1. **测试脚本：** 编写自动化测试代码（如使用 Pytest, Go Test 等）。
2. **边界覆盖：** 必须包含对接口超时、厂商限流、非预期 Json 响应等异常链路的自动化断言。

## 3. 输入规范 (Input & Mocking)

请在本地 Mock 出以下场景用于验证你的系统：

* **模拟厂商 A：** 设定为不稳定节点（例如：30% 概率返回 `429`，20% 概率返回 `500`）。
* **模拟厂商 B：** 设定为稳定但慢速节点（例如：恒定响应 `200`，但延迟在 2s 以上）。

## 4. 交付物规范

请提交包含以下内容的压缩包或代码库：

1. **README 文档：** 包含调研报告、方案流程图（Mermaid）、一键启动说明及 API 调用示例。
2. **源代码：** 逻辑清晰、具备工程化规范的代码包。
3. **测试报告：** 提供一份测试运行结果（截图或 log），重点展示在 **Mock 不同故障情况**下，系统是如何成功切换并记录数据的。

## 5. 评分维度 (Scorecard)

| 维度 | 考察要点 |
| --- | --- |
| **调研深度** | 是否能准确识别现有方案的痛点，设计目标是否明确。 |
| **工程质量** | 代码结构是否解耦（如增加厂商 C 的难度）、是否有竞态处理、日志是否规范。 |
| **容错机制** | Fallback 逻辑是否严密？重试策略是否考虑了对下游的压力控制？ |
| **可观测性** | 成本与成功率统计是否准确，数据结构设计是否合理。 |
| **自动化意识** | 测试用例是否真正覆盖了异常场景，而非仅测试正向链路。 |
