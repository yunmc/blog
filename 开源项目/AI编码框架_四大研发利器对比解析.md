# Vibe Coding 四大 AI 研发框架深度对比与融合

在当前 AI 辅助编程领域（Vibe Coding）中，如何约束和编排大语言模型（AI Agent）进行稳定、可靠的开发是一个核心命题。

现存的四大研发利器（BMAD、gstack、OpenSpec、Superpowers）分别代表了控制 AI 的四个不同维度。

如果用“开公司建大楼”来打比方：
*   **BMAD** 是**《公司研发管理制度》**（按静态文档驱动的宏观方法论与协同理念）。
*   **gstack** 是你雇来的**“外籍专家施工团队”**（强依赖显式指令如 `/pm` 召唤 CEO、架构师、工人的虚拟角色流线）。
*   **OpenSpec** 是你和施工队伍签订的**“严谨工程施工单次合同”**（轻量级、强制无歧义的显式需求契约）。
*   **Superpowers** 是赋予每位工人的**“超能力植入体/被动工作本能”**（安装后自动生效的工作习惯，如自动进行交互式需求访谈、实施 TDD 测试驱动开发、切分任务给 Subagent 执行）。

## 1. 架构与定位维度横向对比

| 项目 | 核心定位 | 运作机制与特征 | 最适用场景 |
|------|--------------|---------|---------|
| **BMAD** | **结构化敏捷 AI 研发方法论** | **宏观抽象体系**：采用大量的 `.md` 文档（如 ADR 架构决策、开发计划）限制 AI 输出边界，主张 Planner 和 Coder 的多 Agent 协同。 | 需要防止架构腐化、管理庞杂上下文的**中大型复杂项目防偏航长期迭代**。 |
| **gstack** | **全栈角色工作流管道** (Role-Playing) | **硬编码框架包**：内置 18 种专家角色，在 CLI 手动使用斜杠命令 (如 `/pm`, `/dev`) 强制召唤，并驱动 AI 走完 SDLC 软件生命周期。 | 极其适合**全栈开发者需要明确把控每个研发环节**，或从 0 到 1 破荒搭设 MVP 的项目。 |
| **OpenSpec**| **轻量级需求契约框架** (Spec-Driven) | **具体显式契约**：严格执行 `Proposal -> Apply -> Archive`。要求开发者用标准语法写出 `proposal.md`，AI 按明确的 `SHALL` 要求办事。 | 日常迭代中，经常因为 **AI 遗嘱需求细节、改出代码 Bug**，需要一板一眼精准控制单次交付的场景。 |
| **Superpowers** | **被动式工程本能与技能集** (Agentic Skills Framework) | **全自动交互防线**：安装即生效的基础指令集。会在你布置任务时**主动拉停**并反向采访你确认需求，随后自动展开基于 TDD (测试驱动测试) 与 YAGNI 原则的代码编写自查循环。 | 适合任何想让 AI **具有独立工程师素养（而非打字员）**的基础环境，不再需要开发者像保姆一样盯着输出。 |

## 2. 深度视角：约束权来源与人机交互模式

### 2.1 它们是如何控制 AI 的？
*   **BMAD（靠“环境”控制）**：不手把手教 AI 怎么做，而是预置好 Markdown 文档（ADR 规划、架构说明）画好圈子。AI 介入时必须先遵守这套环境法则。*(哲学：与其管每次输出，不如建好优质上下文)*
*   **gstack（靠“角色”控制）**：用特定身份压制 AI。例如敲下 `/qa` 强制要求其输出逻辑收束到测试阶段规范中，各司其职。*(哲学：用专业角色边界强制分解研发管线)*
*   **OpenSpec（靠“契约”控制）**：不在乎 AI 扮演什么角色，只给它一份带有 `SHALL` 和 `GIVEN..WHEN..THEN` 的冰冷功能验收合同。*(哲学：代码只是实现，消除歧义的规范才是真理)*
*   **Superpowers（靠“本能”改造）**：直接重塑 AI 的“性格”。当你丢一个模糊需求过去时，它会主动拉停你并反向采访确认，再自行启动 TDD 循环。*(哲学：直接把 AI 升级成能主动思考的高级工程师)*

### 2.2 开发者交互状态对比（谁更累？）
| 框架 | 开发者的工作状态特征 | 疲劳与心智负担点 |
| :--- | :--- | :--- |
| **BMAD** | **“架构总监”模式**：主要负责写 `.md` 规划方案，不碰具体实现。 | 极耗结构化脑力，如果系统文档没写透，容易带歪后续工程。 |
| **gstack** | **“流水线包工头”模式**：终端高频输入 `/pm`, `/dev` 交接。 | 需要熟悉 SDLC 各环节链路才知道下一步该 `@` 哪个角色。 |
| **OpenSpec** | **“甲方验收员”模式**：在 `proposal.md` 写“如果...那么...必须”的契约。| 枯燥繁琐，写严谨 Spec 规范的门槛有时候堪比重写一遍业务代码。 |
| **Superpowers** | **“平级合作同事”模式**：定大方向，做 AI 抛回来的选择题。 | AI 有时会过分“轴”于边界，控制欲极强的开发者会觉得它啰嗦。 |

**📌 一句话灵魂总结（开发一个购物车）：**
*   **BMAD** 说：“请查阅根目录的 `ADR-03-前端规范`，按那里说的写。”
*   **gstack** 说：“`/pm` 先出需求，`/dev` 接着写视图，最后 `/qa` 上测试！”
*   **OpenSpec** 说：“按 `proposal.md` 搞定这 3 个断网测试用例，全绿再找我！”
*   **Superpowers** (AI主动发话)：“了解。但断网时怎么防重扣？我先加几个防御用例，同意的话我就开始！”

## 3. 四大利器如何融合使用 (Vibe Coding 黄金管线)

在当今的 AI 编程实践中，高阶开发者可以通过“套娃”的方式将这四个思想和工具完美融合：

1. **基础赋能（Superpowers）**
   配置你的 AI 环境（如 Cursor / Claude Code），安装 **Superpowers** 赋予 AI 工程本能（让它学会自己写测试、自查代码）。
2. **建立公约（BMAD）**
   在项目根目录设立符合 **BMAD** 理念的 Markdown 文档矩阵基约，比如建立 `/docs/ADR/` 目录，让具有本能的 AI 熟悉整体项目的宏观规则。
3. **极速开荒（gstack）**
   在新设立项目模块或 MVP 起步时，手动使用 **gstack** 的 `/architect` 和 `/ceo` 角色极速跑通需求定义和系统骨架的生成。
4. **攻坚交付（OpenSpec）**
   当骨架建立完毕进入日常的复杂业务功能迭代时，要求每一次 Feature 开发都经过 **OpenSpec** 编写一份验收标准清晰的 `proposal.md`。使用 `openspec apply` 让具有 Superpowers 技能的 AI 严格按契约交付，随后 `archive`。

通过上述“底层赋能+宏观规章+快速建队+严谨执行”的矩阵，开发者将真正从“打字员”转变为“研发合伙人”和“架构总监”。
